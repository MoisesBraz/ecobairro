**Resposta de OP3 (fila de prioridades):**

```
[
  {
    ecoponto_id: uuid,
    nome: string,
    localizacao: { lat, lng },
    zona_nome: string,
    nivel_enchimento: integer,
    estado: string,
    ultima_leitura_em: ISO8601,
    minutos_sem_leitura: integer,
    score_prioridade: float,   ← calculado: nivel * peso_nivel
                                  + minutos_offline * peso_tempo
    alertas_abertos: integer
  },
  ...
]
```

O `score_prioridade` é calculado no FastAPI com pesos configuráveis por zona. Ecopontos em `OFFLINE` há mais de 2× o `timeout_offline_min` têm penalização adicional de prioridade — não significa que estão cheios, mas que precisam de verificação.

**Resposta de OP4 (sugestão de rota) — implementado:**

```
{
  motor: "osrm" | "greedy",         ← "greedy" = OSRM indisponível (linha reta)
  zona: string | null,
  distancia_m: float, duracao_s: float,
  distancia_label: string,          ← ex. "8.2 km"
  duracao_label: string,            ← ex. "1h 45min"
  paragens: [ { id, nome, lat, lng, ocupacao, ordem } ],   ← ordem de visita
  geometria: [[lat, lng], ...]      ← traçado por estradas (OSRM) ou linhas retas
}
```

Seleção: ecopontos ativos com `ocupacao >= limiar` (default 60) **ou** sensor `offline`/`alerta`,
ordenados pelo score da fila OP3, com cap de 12 paragens. O motor primário é o **OSRM `/trip`**
(`OSRM_BASE_URL`, default público `router.project-osrm.org`); em falha cai para um fallback
greedy (vizinho-mais-próximo) — nunca devolve dados mock. O `GET` é dinâmico (depende da posição
do veículo) e **não** tem cache. O resultado é gravado pelo gestor via `POST /v1/rotas`.

> OP5 (aceitar), OP6 (concluir com `rotas_execucao`) e OP7 (export CSV/XLSX) continuam **futuro**.
> A persistência atual reutiliza a tabela `rotas` (campos `geometria`/`paragens`/`zona` em JSON).
