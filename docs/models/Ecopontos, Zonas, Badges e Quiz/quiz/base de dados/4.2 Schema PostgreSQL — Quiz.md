
> **Nota de implementação (atual).** O backend usa estas tabelas mas com um
> design de **banco + sorteio**: a tabela `quizzes` guarda um *pool* ativo
> ("Banco de Perguntas EcoBairro", seed em `apps/api/prisma/quiz-bank.ts`) e o
> serviço **sorteia `numero_perguntas` perguntas aleatórias** (Fisher-Yates) por
> sessão, em vez de servir um quiz fixo. As invariantes mantêm-se (exatamente 1
> opção `correta`, `correta` nunca no payload, feedback educativo sempre). Não há
> geração por LLM/ML em runtime. Endpoints implementados em
> `/v1/gamification/*` (ver `apps/api/src/gamification/CLAUDE.md`); ranking
> anónimo por zona fica para futuro.
>
> **Gestão de perguntas (GESTOR/ADMIN) implementada** — sem alteração de schema:
> `GET/POST/PATCH/DELETE /v1/admin/quiz/perguntas` faz CRUD de `quiz_perguntas` +
> `quiz_opcoes` no pool ativo. As perguntas passam a ser **editáveis em runtime**
> (já não só via seed). O seed é agora *create-if-absent* (não apaga edições do
> gestor). Ver [[models/Ecopontos, Zonas, Badges e Quiz/quiz/Gestão (admin)]].

A tabela `quiz_sessoes` foi definida no documento do domínio Cidadão:

```
quiz_sessoes
  id              UUID   PK
  cidadao_id      UUID   FK → users(id)
  quiz_id         UUID   FK → quizzes(id)
  tipo            VARCHAR(10) CHECK IN ('SEMANAL','DIARIO')

  score_obtido    INTEGER
  total_perguntas SMALLINT

           (cidadao_id, tipo, concluido_em DESC)
           (quiz_id, concluido_em DESC) — para Q9 estatísticas e Q12 verificação
```

**Regra "exactamente uma opção correcta" — três camadas:**

```
Camada 1 — NestJS validator no Q11 (criar quiz):
  respostas.filter(o => o.correta).length === 1
  senão → 400 Bad Request com detalhe por pergunta

Camada 2 — Índice parcial de apoio ao scoring:
  INDEX ON quiz_opcoes(pergunta_id) WHERE correta=true
  → NestJS consulta este índice para pontuar Q4
    em vez de SELECT * e filtrar em memória

Camada 3 — Validação antes de publicar (Q11 transacção):
  Para cada pergunta: COUNT(quiz_opcoes WHERE correta=true AND pergunta_id=?)
  Se qualquer pergunta tiver ≠ 1: rollback da transacção completa
```

**Regra crítica — campo `correta` nunca em cache:** O payload guardado em `quiz:atual:SEMANAL` e `quiz:{id}:perguntas` é construído pelo NestJS antes de guardar em Redis, com o campo `correta` removido de todas as opções. A consulta da opção correcta ocorre apenas no endpoint Q4, directamente no PostgreSQL via índice parcial `WHERE correta=true`, nunca passando pelo Redis.
